## Why

GKIM 现在已经有可交付的 Android 客户端，但仓库里还没有任何 GitHub CI/CD 流程来在打版本 tag 后自动产出 APK。每次发布都依赖本地手工构建，既不稳定，也缺少可追溯的版本产物和统一的发布入口。

## What Changes

- Add a GitHub Actions workflow that triggers when a version tag is pushed and builds the Android release APK on GitHub-hosted runners.
- Normalize tag-driven release metadata so the generated APK and GitHub Release asset clearly correspond to the pushed version tag.
- Introduce secret-managed release signing inputs for CI so the workflow can publish a distributable APK without committing keystores or passwords into the repository.
- Upload the built APK both as a workflow artifact and as a tag-bound GitHub Release asset so reviewers and testers can download the exact build generated by CI.
- Document the required repository secrets, supported tag format, and operator workflow for creating an Android release from GitHub.

## Capabilities

### New Capabilities
- `android-release-automation`: GitHub tag-triggered Android APK build, signing, and release publication for the GKIM client.

### Modified Capabilities
- None.

## Impact

- Affected code: `.github/workflows/**`, Android Gradle release/versioning configuration under `android/`, and release/operator documentation in `README.md` plus Android-facing docs as needed.
- Affected APIs: GitHub Actions tag-triggered workflow execution, GitHub Release asset upload, and Gradle properties consumed by CI.
- Affected dependencies: GitHub-hosted runners, JDK 17, Android SDK / build tools for API 34, Gradle caching, and secret-managed signing material stored in GitHub repository settings.
- Affected systems: Android release packaging path, repository release process, and the operator workflow used to cut a downloadable APK build from a Git tag.
