# GKIM 项目技术与选型说明

## 1. 项目定位

GKIM 是一个面向移动端场景的 IM 应用项目，目标不是只做一个“能发消息的 Demo”，而是围绕真实可用的即时通信体验，完成一套从客户端到服务端的完整闭环：

- 支持注册、登录和安全会话持久化
- 支持用户搜索、好友请求、联系人关系建立
- 支持基于联系人关系的 1 对 1 实时消息
- 支持离线后重新拉取历史、未读和会话状态恢复

从产品形态上看，它是一个移动端 IM App；从工程视角看，它更像一个完整的全栈通信系统验证项目。

## 2. 技术总览

| 层级 | 技术 | 作用 |
| --- | --- | --- |
| Android 客户端 | Kotlin、Jetpack Compose、Navigation Compose | 构建原生移动端 UI、导航和交互层 |
| Android 状态与数据层 | ViewModel、Repository、Coroutines | 做界面状态管理、业务编排和异步流程控制 |
| Android 本地能力 | Room、DataStore、EncryptedSharedPreferences | 提供本地数据持久化、配置存储和安全会话存储 |
| Android 网络层 | Retrofit、OkHttp、Kotlinx Serialization、WebSocket | 对接后端 HTTP API 和实时消息通道 |
| 后端服务 | Rust、Axum、Tokio | 构建高并发、异步的 HTTP + WebSocket IM 服务 |
| 后端数据层 | PostgreSQL、SQLx | 持久化用户、好友关系、会话、消息、未读状态 |
| 安全能力 | Argon2 | 用于密码哈希，支撑账号注册与登录 |
| 可观测与运维 | tracing、Docker、Ubuntu systemd 流程 | 支撑本地验证、容器化运行和服务部署 |
| 工程规范 | OpenSpec、自动化测试、验收记录 | 用规范驱动实现、验证和交付过程 |
| 早期原型资产 | UniApp、Vue3、Pinia、UnoCSS | 用于前期跨端原型、设计探索和信息架构沉淀 |

这个项目最核心的主线技术组合，是 `Android 原生客户端 + Rust IM 后端 + PostgreSQL + WebSocket`。

## 3. 为什么这样选型

### 3.1 为什么客户端选择 Android 原生，而不是继续停留在跨端原型

仓库里保留了早期 UniApp / Vue3 原型，这说明项目一开始是以“快速验证界面和交互结构”为目标推进的。但在真正进入 IM 产品能力建设后，我把主交付路线收敛到了 Android 原生，原因很明确：

1. IM 应用对交互流畅度、页面状态管理、生命周期、实时连接管理的要求很高，原生方案更稳定。
2. Jetpack Compose 可以更自然地组织复杂 UI 状态，配合 Navigation Compose、ViewModel、Repository，能形成清晰的分层结构。
3. 原生 Android 对本地存储、安全会话、测试设施、模拟器验证链路支持更成熟，更适合做“真实可用”而不是“只展示界面”的移动应用。

换句话说，早期跨端方案适合验证方向，原生方案更适合把 IM 产品能力真正做深、做稳。

### 3.2 为什么后端选择 Rust

Rust 是这个项目非常重要的技术决策之一，它不是为了“炫技”，而是因为 IM 后端天然具备几个特点：高并发、长连接、异步事件流、状态一致性要求高。

在这个项目里，我选择 Rust 后端，主要基于以下判断：

1. **内存安全和并发安全更强**  
   即时通信服务会同时处理 HTTP 请求、WebSocket 长连接、消息广播和数据库读写。Rust 在语言层面避免了很多常见的内存错误和数据竞争问题，能显著降低高并发服务中的隐患。

2. **适合构建异步、高吞吐的实时服务**  
   这里采用的是 `Axum + Tokio` 组合：  
   - `Axum` 负责 HTTP API 和 WebSocket 升级入口  
   - `Tokio` 负责异步运行时和任务调度  
   这套组合很适合实现 IM 场景里的“登录鉴权 + 实时推送 + 状态回放”一体化服务。

3. **便于把实时和持久化逻辑做成一套清晰的服务边界**  
   在 GKIM 中，Rust 后端不仅负责用户注册、登录、好友请求等 REST API，还负责 WebSocket 实时事件通道，以及 PostgreSQL 中消息、会话、未读状态的恢复逻辑。  
   这意味着后端不是一个简单的 API 中转层，而是真正承接业务状态和通信时序的核心。

4. **工程表达更偏“系统能力建设”**  
   对求职来说，Rust 后端本身也能很好体现我在系统设计、并发模型、安全性和服务边界建模方面的能力，而不仅仅是“会调用框架”。

### 3.3 为什么选择 PostgreSQL + WebSocket 的组合

这个组合的本质是把“实时性”和“可靠性”拆开处理：

- `WebSocket` 负责在线时的低延迟消息推送、送达回执、已读回执、好友请求事件
- `PostgreSQL` 负责离线后的消息历史、未读计数、会话恢复、联系人关系和好友请求状态持久化

这样设计有两个直接好处：

1. 在线时用户能获得接近实时通信的体验
2. 离线、重连、重启后，系统仍然可以从数据库恢复出一致状态，而不是依赖内存中的临时数据

对于 IM 来说，这比“只做 WebSocket 推送”更接近真实产品形态。

### 3.4 为什么强调 OpenSpec 和测试驱动的交付过程

这个项目不只是“把功能写出来”，还把规格、验证和验收串成了闭环。

我在仓库中引入了 OpenSpec 变更流和任务验收记录，配合：

- Rust 集成测试
- Android 单元测试
- Android 仪器测试
- 验收证据与质量分数记录

这样做的价值在于：每一项功能不是“感觉完成了”，而是有规格、有验证命令、有结果记录、有同步到主规范的过程。这对多人协作、复杂功能演进和长期维护都非常重要。

## 4. 项目中的关键工程亮点

### 4.1 账号体系不只是页面，而是完整链路

项目已经打通了从 `注册 / 登录` 到 `Token/Session 持久化` 再到 `启动时会话恢复` 的完整流程。  
这意味着客户端、后端和本地安全存储是联动设计的，而不是只做了两个表单页面。

### 4.2 好友关系作为消息权限边界

我没有把消息系统做成“任何人都能直接发消息”的弱约束模型，而是加入了：

- 用户搜索
- 好友请求发送/接受/拒绝
- 只有互为联系人后才能建立真实消息链路

这使系统更接近真实社交 IM，而不是简单聊天室。

### 4.3 后端既有 REST 接口，也有实时网关

后端同时承担两类职责：

- 基于 HTTP 的注册、登录、搜索、好友请求、会话引导接口
- 基于 WebSocket 的实时消息、送达、已读和关系事件推送

这套设计体现了我对“请求-响应模型”和“事件驱动模型”并存的理解，而不是只会做 CRUD。

### 4.4 强调状态恢复，而不是只强调在线发送

项目里不仅验证了在线收发消息，也覆盖了：

- 消息历史拉取
- 未读状态重建
- 离线后重连恢复
- 启动后重新 bootstrap 会话

这类能力是 IM 产品从 Demo 走向可用系统的关键分水岭。

### 4.5 自动化测试覆盖真实交付链路

测试不是装饰性的。这个项目中，验证链路覆盖了：

- Rust 后端的 schema / API / WebSocket 集成测试
- Android 的 Repository / ViewModel 单元测试
- Android 登录、注册、会话恢复、联系人流转的仪器测试

这说明我不仅关注功能实现，也关注可验证性和回归成本。

- 在数据与通信层，我用 PostgreSQL + WebSocket 组合平衡实时性与可靠性
- 在工程流程上，我用 OpenSpec 和自动化测试保证实现不是“看起来完成”，而是“真正可以被验证和交付”

如果把它放到求职材料中，我认为它最有价值的地方，是它能够证明我不仅能写功能，还能设计系统、做技术取舍，并把复杂功能稳定地落到工程实践里。
